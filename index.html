クラウド設計・マネージドサービス・移行パターン

Section 01

オンプレ vs クラウド

システムを「所有する時代」から「利用する時代」へ

オンプレとクラウドの本質的な違い

🏢 オンプレミス

「所有」

  • 設備・機材を自前で用意し設置して利用
  • 2010年前半まで主流
  • 全て自社でコントロール可能
☁️ クラウド

「利用」

  • SaaS / PaaS / IaaS として提供されるものを活用
  • 現在はクラウド／ハイブリッドが主流
  • 使った分だけ支払い
世の中のシステムは 「オンプレ」か「クラウド」 のどちらかで稼働している。
両者の本質的な違いは 所有 vs 利用 の考え方にある。

オンプレ → クラウド移行が増える理由

🏗️ インフラの制約

  • 設備設置スペースが必要
  • CPU は後から変更不可 → 追加費用で買い直し
  • トラフィック増加への対応が困難
  • ネットワーク機器の専門知識が必要（CCNA 等）

⚙️ 運用の重さ

  • 物理故障 → 機器交換が必要
  • 設備のメンテナンス・監視要員が必要（24/7）
  • 設備廃棄は数千万円〜億円単位のコスト
  • 非機能要件を物理設備の設計に落とし込む複雑さ
⚠️ 冗長化コストが爆発する：サーバー数十〜数百台調達、UPS・発電設備、複数系統の電源・回線引き込み…
SLA を 99% → 99.9% → 99.99% と高めるたびに指数関数的にコストが増加。

オンプレ vs クラウド — 比較

🏢 オンプレ ☁️ クラウド
初期コスト 極めて高い（数千万円↑） 0 円（Pay-as-You-Go）
スケーラビリティ 手動・物理的制約 自動・論理的拡張
調達速度 遅い（設置まで含むと数週間） 速い（数分〜数時間）
障害対応 自前・フルスタック サービス依存・抽象化
（責任共有モデルに基づく）
Section 02

クラウド設計の考え方

Design for Failure / Well-Architected Framework

クラウド設計とは？

「動くものを作る」 から 「変化に強い仕組みを作る」 へ
🏚️ 従来（オンプレ）の設計思想

ピーク時に耐えられる 「最大サイズ」 を固定で設計する

→ 常時オーバースペックで無駄が多い

☁️ クラウドの設計思想

需要に合わせて 「伸び縮み」 し、
失敗しても 「すぐにやり直せる」 設計

→ 無駄ゼロ・回復力重視

💥 Design for Failure

「サーバーはいつか必ず壊れる」と考え、
1台が死んでもサービスを止めない仕組みを最初から組み込む

🔗 疎結合（Loose Coupling）

DB・認証・アプリを独立させ、一部の変更が全体に影響しない構造にする

クラウド設計の5大原則

AWS Well-Architected Framework の視点

柱 ポイント
⚙️ 運用性 手作業を減らし、コードでインフラを管理する（IaC）
🔒 セキュリティ 全レイヤーで防御し、権限は 「最小限」 だけ与える
✅ 信頼性 障害からの 自動復旧 と、データの 多重化
⚡ パフォーマンス マネージドサービスを使い、負荷に応じて 自動拡張 する
💰 コスト最適化 不要なリソースを止め、使った分だけ払う 仕組み
Section 03

マネージドサービス

「餅は餅屋へ」— インフラ管理を手放す

マネージドサービスとは？

クラウドベンダーがインフラの 構築・運用・保守 を「肩代わり」してくれるサービス。
利用者は 「アプリ開発」 だけに専念できる。
😓 セルフマネージド（自前）

利用者がやること：

  • OS パッチ当て
  • バックアップ設計・実行
  • 冗長化構成
  • セキュリティ更新
  • ハード故障対応
  • キャパシティ設計

= 膨大な「作業」

😊 マネージドサービス

利用者がやること：

  • データ投入
  • コード実行
  • 設定の最適化

ベンダーがやること：
上記の「作業」をすべて担う

マネージドサービスの価値

「守り」の自動化と「攻め」の高速化

メリット
⚡ スピード 数クリック・数分でエンタープライズ級の環境が整う
✅ 信頼性 自動バックアップ、マルチ AZ 配置による高可用性が標準
📈 スケーラビリティ アクセス増に応じて自動拡張（オートスケール）
💰 コスト最適化 使った分だけの従量課金

クラウドネイティブな設計への進化

「サーバー」を意識しない設計（Serverless & Managed）

設計者の役割が変わる：
「メモリやディスクの計算」 → 「どのサービスとどのサービスを組み合わせると最大にビジネス価値を生み出せるか」
（サービスのオーケストレーション）

マネージドサービスの思想

「餅は餅屋へ」 — 付加価値を生まない作業からの脱却
🚫 Undifferentiated Heavy Lifting

設備調達・メンテ・物理障害対応は
ビジネスの成功に直接貢献しない

→ これらをクラウドベンダーに丸投げし、
自社リソースを「独自の価値」に集中させる

⚖️ 責任共有モデルの理解

インフラの可用性・セキュリティの「土台」はベンダーが保証

ユーザーは「その上のデータと設定」に責任を持つ

Section 04

責任共有モデル

「物理」はベンダー、「データ」は自分

責任共有モデルとは？

「どこまでがクラウド側の責任で、どこからが自分の責任か」 を明確にするルール
☁️ ベンダーの責任

「クラウドのセキュリティ」
（インフラ自体）

  • データセンターの物理保護
  • ハードウェア・ネットワーク基盤
  • 仮想化レイヤー
👤 ユーザーの責任

「クラウド内のセキュリティ」
（データと設定）

  • データの暗号化
  • アクセス権限の設定
  • ファイアウォール設定
⚠️ 責任の所在が曖昧だと → パッチ当ての漏れ・設定ミスによる 情報漏洩 に繋がる

責任の境界線

担当箇所 ☁️ ベンダー責任 👤 ユーザー責任
インフラ データセンターの物理保護、サーバーの故障交換 —
ネットワーク 物理ネットワークの維持、仮想化基盤の防御 ファイアウォールの設定、通信の暗号化
OS・ソフト マネージドサービスの場合のパッチ当て 仮想サーバー（EC2等）の OS アップデート
データ — データの暗号化、バックアップ、権限管理

サービス形態による責任範囲の変化

マネージドサービスを使うほど、ユーザーの「責任（苦労）」は減る

☁️ ベンダー管理
👤 ユーザー管理
オンプレ
空調・電源・HW・ネットワーク・OS・ミドルウェア・アプリ・データ（すべて自分）
IaaS
空調・電源・HW・ネットワーク・仮想化
OS・ミドルウェア・アプリ・データ
PaaS / Managed
〜 OS・ミドルウェアまで
アプリ・データ・設定
SaaS
〜 アプリまでほぼすべて
ID・データのみ
💡 ビジネスに集中したいなら、責任範囲をベンダーに委ねるほど良い
→ より多くのマネージドサービスを積極的に活用する設計を目指すべき
Section 05

クラウド移行パターン（6R）

移行の目的に合わせて最適な手法を選ぶ

クラウドへの移行パターン（6R）

クラウド移行は単にサーバーを移すことではなく、ビジネスの目的（コスト削減・スピード向上・拡張性など）に合わせて最適な手法を選択するプロセス。
🚀 Rehost

そのまま移す

⚙️ Replatform

少し最適化して移す

🔧 Refactor

抜本的に作り直す

🛒 Repurchase

別製品に乗り換える

🗑️ Retire

廃止する

🛡️ Retain

維持（移行しない）

6R-1 Rehost — 「そのまま」持っていく

最短・最速のクラウド化。OSやアプリを構成変更せず、そのままクラウド（EC2等）へ移設。
✅ メリット
  • 移行スピードが最も速く、リスクが低い
  • データセンター撤去期限がある場合に最適
🎯 使い所
  • 複雑すぎて中身をいじれないレガシーシステム
  • まずクラウドに乗せてから最適化したい場合

6R-2 Replatform — 「少しだけ」変える

コアを変えずに恩恵を受ける。アプリの根幹は変えず、DB等をマネージドサービス（RDS等）に置き換える。
✅ メリット
  • OS・DB のパッチ当て、バックアップ運用から解放
  • アプリ改修を最小限に抑えつつクラウドの恩恵を享受
🎯 使い所
  • 運用の手間を減らしたいが
  • アプリを大規模改修する予算や時間がない場合

6R-3 Refactor — 「作り変える」

クラウドネイティブの真髄。クラウドの機能をフル活用するため、
アプリの構造（マイクロサービス化・サーバーレス化）を抜本的に見直す。
✅ メリット
  • スケーラビリティ・耐障害性・コスト効率が最大化
  • ビジネスの変化に極めて強い基盤になる
🎯 使い所
  • 今後も継続的に機能追加が必要な
  • 自社の競争力の源泉となる戦略的システム

6R-4 Repurchase — 「買い替える」

所有から利用への完全転換。既存ソフトを捨て、SaaS（Salesforce, Microsoft 365 等）へ乗り換える。
✅ メリット
  • サーバー管理・アプリ開発が不要になる
  • 常に最新の機能が提供される
🎯 使い所
  • 会計・人事・メールなど
  • 自社で独自開発する必要がない
    「非競争領域」の業務システム

6R-5 Retire — 「捨てる」

資産の整理整頓。移行のタイミングで利用率が低いシステムや不要な機能を廃止する。
✅ メリット
  • 移行コストと将来の運用保守コストをゼロに
  • IT 資産のブラックボックス化を防ぐ
🎯 使い所
  • 数ヶ月間アクセスがないシステム
  • 代替手段がすでにある古いツール

6R-6 Retain — 「維持する」

あえて動かさない勇気。調査の結果、現時点ではクラウドへ移行せずオンプレミスに継続して残す。
✅ メリット
  • 不必要な移行コスト・リスクを回避できる
🎯 使い所
  • 減価償却が終わっていないハードウェア
  • 特殊な周辺機器との接続が必要なもの
  • 法規制でデータ持ち出しができないもの

まとめ

💡 クラウド設計の本質

変化に強い仕組みを作る。
失敗を前提に設計し、自動回復させる。

🔧 マネージドサービス

差別化につながらない重労働をベンダーに委ね、
本来の価値創出に集中する。

⚖️ 責任共有モデル

「物理」はベンダー、「データ・設定」は自分。
境界線を明確に理解する。

🚀 6R 移行戦略

移行はゴールではなくスタート。
ビジネス目的に合った手法を選ぶ。
