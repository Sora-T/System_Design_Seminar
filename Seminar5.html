Section 01

キャッシュの基本

なぜキャッシュが必要か

キャッシュとは？

低速なストレージへのアクセスを減らすため、高速なストレージに結果を一時保存する仕組み。
「同じ計算・同じ問い合わせを繰り返さない」ことでシステム全体の応答速度とコストを改善する。
Cache Hit

キャッシュにデータが存在 → そのまま返す。DBアクセスなし。

→ 高速・低コスト

Cache Miss

キャッシュにデータがない → DBから取得してキャッシュに格納。

→ 初回は遅い

TTL（有効期限）

キャッシュの生存時間。TTL 切れで自動削除され、次回アクセス時に再取得。

整合性とのトレードオフ

ヒット率（Hit Rate）が低いキャッシュは逆効果になりうる。
アクセス頻度が高く、データの変化が少ないものがキャッシュに向いている。

キャッシュの階層

キャッシュはシステムの複数の層に存在する。ユーザーに近いほど速く、容量は小さい。
ユーザーに近い ↑
ブラウザキャッシュ HTTP Cache-Control / ETag
CDN
エッジキャッシュ CloudFront, Fastly, Cloudflare
アプリ層
アプリケーションキャッシュ Redis, Memcached（インメモリ）
DB層
DBキャッシュ / クエリキャッシュ Buffer Pool, Query Cache
↓ データに近い
ストレージ（HDD / SSD） 最終的な永続化先
Section 02

CDN / アプリ / DB キャッシュ

各層のキャッシュの役割と使い所

CDN キャッシュ

Content Delivery Network。世界中のエッジサーバーにコンテンツをキャッシュし、
ユーザーの近くから配信することでレイテンシを削減する。

向いているもの

  • 静的ファイル（JS / CSS / 画像 / 動画）
  • 更新頻度が低いHTMLページ
  • APIレスポンス（パブリックで変化が少ないもの）

向いていないもの

  • ユーザーごとに異なるレスポンス（認証済みコンテンツ）
  • リアルタイム性が求められるデータ

仕組み

ユーザー
↓
エッジサーバー（POP）
Hit → そのまま返す
↓ Miss 時のみ
オリジンサーバー
取得してエッジにキャッシュ

アプリケーションキャッシュ（Redis / Memcached）

DB の前段にインメモリのキャッシュ層を置き、頻繁にアクセスされるデータをメモリ上に保持する。
DBへのクエリ数を劇的に削減できる。
Redis
  • String, Hash, List, Set, Sorted Set など多様なデータ型
  • TTL・パブサブ・Lua スクリプト対応
  • 永続化オプションあり（RDB / AOF）
  • セッション、ランキング、レート制限にも使われる
Memcached
  • シンプルな Key-Value のみ
  • マルチスレッドで高スループット
  • 永続化なし・シンプルな用途向け
  • 水平スケールが簡単
どちらを選ぶか：データ構造の複雑さや永続化が必要なら Redis、
シンプルな KV キャッシュを大量に捌くだけなら Memcached。

DB キャッシュ

Buffer Pool（バッファプール）

RDBMS がディスクのデータページをメモリ上に保持する内部キャッシュ。
同じページへの再アクセスはディスク I/O をスキップできる。
InnoDB buffer pool size を適切に設定することが DB チューニングの基本。

クエリキャッシュ（非推奨）

MySQL の旧機能。同じ SQL のレスポンスをキャッシュしたが、
テーブル更新時に全invalidationが必要で逆効果になりやすく、MySQL 8.0 で廃止。

マテリアライズドビュー

複雑な集計クエリの結果をテーブルとして事前に保存する。
定期的に更新（リフレッシュ）することで、毎回の重いクエリを回避できる。

⚠️ リアルタイムではない。更新頻度とデータの鮮度はトレードオフ。

Read Replica の活用

レポート・分析クエリを Replica に向けることで、
Primary への書き込み負荷を隔離できる。

Section 03

キャッシュ戦略パターン

Read-Through / Write-Through / Write-Behind / Cache-Aside

Cache-Aside（Lazy Loading）

アプリが自分でキャッシュを制御するパターン。最も広く使われる。

読み取りフロー

① アプリ → キャッシュを確認
Hit ↓
キャッシュから返す
Miss ↓
② DB から取得
↓
③ キャッシュに保存
↓
④ アプリへ返す

特徴

  • 実際に要求されたデータだけキャッシュされる（無駄が少ない）
  • キャッシュ障害時もDBに直接アクセスできる（耐障害性◎）
  • 初回アクセス時は必ずキャッシュミスが発生（Cold Start）
⚠️ 書き込み後にキャッシュを明示的に削除 or 更新しないと古いデータを返し続ける。

Read-Through

キャッシュ層自体がDBからの読み取りを担うパターン。
アプリはキャッシュにしか問い合わせず、Miss 時の DB 取得はキャッシュ層が自動で行う。
アプリ
↓ 問い合わせ（常にキャッシュへ）
キャッシュ層
Hit → 即返す　Miss → DBへ
↓ Miss 時のみ自動取得
DB

Cache-Aside との違い

Cache-AsideRead-Through
DB アクセスアプリが担当キャッシュ層が担当
コード量多い少ない
柔軟性高いキャッシュ実装に依存

Write-Through

書き込みをキャッシュと DB に同時に行うパターン。
キャッシュは常にDBと同期されるため、データの鮮度が高い。
アプリ → 書き込みリクエスト
↓
① キャッシュに書き込み
↓ 同期
② DB にも書き込み
↓ 両方完了後
完了を返す

メリット / デメリット

  • ✅ キャッシュと DB が常に同期（整合性◎）
  • ✅ 読み取りは常にキャッシュから返せる
  • ⚠️ 書き込みが遅くなる（DB を待つ必要がある）
  • ⚠️ 一度も読まれないデータもキャッシュに入る（無駄）
Read-Through と組み合わせることで
読み書き両方をキャッシュ層に集約できる。

Write-Behind（Write-Back）

キャッシュへの書き込みを先に完了させ、DB への書き込みを非同期で後から行うパターン。
書き込みパフォーマンスを最大化できるが、整合性リスクを伴う。
アプリ → 書き込みリクエスト
↓
① キャッシュに書き込み → 即 完了
↓ 非同期（バッチ or キュー）
② DB に非同期書き込み

メリット / デメリット

  • ✅ 書き込みレイテンシが非常に低い
  • ✅ DB への書き込みをバッチ化してスループット向上
  • ⚠️ キャッシュ障害時にDBに未反映のデータが消える
  • ⚠️ 整合性の保証が難しい
データロストのリスクがあるため、多少のデータ損失が許容できるユースケース（閲覧数・いいね数など）に限定するのが安全。

キャッシュ戦略 — まとめ比較

戦略 読み取り 書き込み 整合性 向いている用途
Cache-Aside アプリが制御 DBのみ（手動 invalidate） △ 汎用。読み取り多めのAPI
Read-Through キャッシュ経由で自動 DBのみ △ 読み取り重視・コード簡略化
Write-Through キャッシュから返す Cache + DB 同期 ◎ 整合性重視・読み書きバランス
Write-Behind キャッシュから返す Cache 先書き・DB 非同期 △〜✕ 書き込みスループット最優先
Section 04

整合性とのトレードオフ

キャッシュの最大の難所

キャッシュ無効化（Cache Invalidation）

コンピュータサイエンスの有名な格言：
「難しい問題は2つしかない。キャッシュの無効化と、名前付けだ」
— Phil Karlton
TTL ベース

一定時間後に自動削除。シンプルだが、TTL 中はデータが古いまま。

許容できる staleness を決めて TTL を設定する。

イベントドリブン

DBが更新されたタイミングでキャッシュを即 削除 or 更新する。

整合性◎だが実装が複雑。更新漏れのリスクがある。

バージョニング

キャッシュキーにバージョン番号を含める。古いキーは自然に使われなくなる。

削除不要だが古いキーがメモリを占有する。

⚠️ Cache Stampede（キャッシュスタンピード）：TTL 切れ直後に大量リクエストが同時に DB へ殺到する現象。
対策：確率的 TTL 延長（Probabilistic Early Expiration）や Mutex ロック。

整合性レベルの選択

強整合性とキャッシュの高速化は基本的にトレードオフ。
ユースケースごとに「どこまで古いデータを許容できるか」を判断する。
整合性レベル 概要 向いている例
強整合性 常に最新データを返す。キャッシュを使わない or 即時無効化。 残高照会・在庫確認・決済
結果整合性 しばらくすれば最新になる。TTL ベースで十分なケースが多い。 SNS タイムライン・商品説明
ゆるい整合性 多少古くても問題ない。長い TTL やバッチ更新で OK。 PV数・いいね数・ランキング
設計の問いは「キャッシュするか否か」ではなく、
「このデータは何秒古くても許容されるか」。

まとめ

キャッシュの基本

低速ストレージへのアクセスを減らす仕組み。
ヒット率が高く変化の少ないデータに有効。

キャッシュの階層

ブラウザ → CDN → アプリ（Redis）→ DB の順。
ユーザーに近いほど速いが、容量は小さい。

書き込み戦略

整合性重視 → Write-Through。
スループット重視 → Write-Behind。
汎用 → Cache-Aside。

整合性とのトレードオフ

「何秒古くても許容できるか」を決めてから設計する。
Cache Stampede 対策も忘れずに。
