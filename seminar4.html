Section 01

SQL vs NoSQL

データの構造と整合性要件で選ぶ

RDBMS（リレーショナルDB）とは

データをテーブル（行・列）で管理し、SQLで操作する。
テーブル間を外部キーで結び、整合性を保つ。

ACID 特性

  • Atomicity — トランザクション全体が成功か失敗か
  • Consistency — 常に整合性のある状態を維持
  • Isolation — 同時実行しても互いに影響しない
  • Durability — コミット済みデータは永続化される

代表的な製品

PostgreSQL

OSS。高機能・拡張性◎

MySQL

OSS。Web系で広く普及

マネージド。運用負荷ゼロ

グローバル分散RDBMS

NoSQLとは

「Not Only SQL」。固定スキーマを持たず、用途に特化したデータモデルで大規模・高速なアクセスを実現する。
Key-Value

シンプルな辞書型。
超高速読み書き。

Redis, DynamoDB

Document

JSON/BSON 形式。
柔軟なスキーマ。

MongoDB, Firestore

Wide Column

列ファミリー構造。
時系列・ログ向き。

Cassandra, Bigtable

Graph

ノード・エッジ構造。
関係性の探索向き。

Neo4j, Amazon Neptune

NoSQL は ACID を犠牲にする代わりに BASE（結果整合性・高可用性・柔軟なスキーマ）を重視する設計が多い。

SQL vs NoSQL — 比較

SQL（RDBMS） NoSQL
データ構造 固定スキーマ（テーブル） 柔軟（JSON、KV、グラフ 等）
整合性 強整合性（ACID） 結果整合性（BASE）が多い
スケール 主に垂直スケール 水平スケールが得意
クエリ SQL で複雑な結合・集計が可能 シンプルなアクセスパターンに最適化
向いてる用途 EC・金融・ERP など整合性最優先 SNS・IoT・キャッシュ・検索 など大規模低レイテンシ
どちらかが「優れている」わけではない。アクセスパターン・整合性要件・スケール規模で選ぶ。

どちらを選ぶか — 判断フレームワーク

SQL を選ぶとき
  • トランザクションの整合性が必須（金融・在庫）
  • 複雑な結合・集計クエリが必要
  • スキーマが安定していて変更が少ない
  • チームに SQL の習熟者がいる
NoSQL を選ぶとき
  • スキーマが頻繁に変わる、または不定形データ
  • 大量の書き込み・水平スケールが必要
  • アクセスパターンが単純（特定キーで取得）
  • 結果整合性で許容できるユースケース
⚠️ 現実のシステムは ポリグロット・パーシステンス — 用途ごとに複数のDBを使い分けるのが一般的。
例）ユーザー情報は PostgreSQL、セッションは Redis、商品カタログは MongoDB
Section 02

インデックス

クエリを速くする仕組みとトレードオフ

インデックスとは？

テーブルの目次。インデックスなしでは全行スキャン（フルスキャン）が必要だが、
インデックスがあると O(N) → O(log N) で検索できる。

B-Tree インデックスの仕組み

30 70
↙　　　↓　　　↘
1020 4060 8090

各ノードがソート済み → 二分探索で高速検索

インデックスが効く場面

  • WHERE 句での等値・範囲検索
  • ORDER BY / GROUP BY のソート回避
  • JOIN のキー結合
  • カバリングインデックス（SELECT 列ごとインデックスに含む）

インデックスの種類

種類 特徴 向いている用途
B-Tree デフォルト。等値・範囲検索・ソートに対応 一般的な WHERE / ORDER BY
Hash 等値検索のみ超高速。範囲検索は不可 = による完全一致検索
複合インデックス 複数列を組み合わせたインデックス 複数条件の絞り込み。列順序が重要
部分インデックス 条件を満たす行だけにインデックスを作成 NULL 除外・特定ステータスのみ
全文検索インデックス テキストのトークン分割・転置インデックス LIKE '%word%' の代替、全文検索

インデックスのトレードオフ

✅ メリット
  • SELECT クエリが劇的に高速化
  • ソート・集計のコスト削減
  • JOIN のパフォーマンス向上
⚠️ コスト
  • INSERT / UPDATE / DELETE が遅くなる
    （インデックスの更新が必要なため）
  • ディスク・メモリ消費が増加
  • 多すぎるインデックスはオプティマイザを迷わせる
インデックスは「読み取りが多く、書き込みが少ない」列に張るのが基本。
EXPLAIN / EXPLAIN ANALYZE でクエリプランを確認し、必要なものだけ作る。
よくある失敗：低カーディナリティ（true/false など）の列にインデックスを張っても効果が薄い。
Section 03

レプリケーション vs シャーディング

DBをスケールさせる2つのアプローチ

レプリケーションとは

同じデータを複数のDBノードにコピーする仕組み。
読み取り負荷の分散と、障害時の自動フェイルオーバーが目的。
Primary
書き込み
レプリケーション →→→
Replica 1
読み取り
Replica 2
読み取り
✅ できること
  • 読み取りスループットの向上
  • Primary 障害時のフェイルオーバー
  • バックアップ・レポート用途をReplicaに分離
⚠️ 注意点
  • 書き込みは Primary 1台に集中（書き込みは増やせない）
  • 非同期レプリケーションはレプリケーション遅延が発生しうる
  • データ量は減らない（全ノードが同じデータを持つ）

シャーディングとは

データを複数のDBノードに水平分割する仕組み。
各ノード（シャード）は全体の一部だけを持つ。書き込みスケールとデータ量の増大に対応できる。
アプリ
shard_key で振り分け
→
Shard A
user_id 0〜33%
Shard B
user_id 33〜66%
Shard C
user_id 66〜100%
⚠️ シャードをまたぐ JOIN・トランザクションは非常に困難。シャードキーの設計が最重要。
偏ったキーを選ぶと ホットスポット（一部シャードに負荷集中）が発生する。

レプリケーション vs シャーディング

レプリケーション シャーディング
目的 読み取り分散・高可用性 書き込み分散・データ量の水平拡張
データの持ち方 全ノードが同じデータをコピー 各ノードがデータの一部のみを保持
書き込みスケール △（Primary 1台が上限） ◎（シャード数に比例して増加）
複雑さ 比較的シンプル 高い（クロスシャードクエリ・リバランス等）
障害対応 ◎（フェイルオーバー自動化が容易） △（特定シャード障害でそのデータが不可）
実際のシステムは両方を組み合わせるのが一般的。
例）各シャードに対して Primary + Replica を構成する。

CAP 定理 — 分散DBの限界

分散システムは C（整合性）・A（可用性）・P（分断耐性） の3つを同時には満たせない。
ネットワーク分断（P）は現実に発生するため、CA か CP か AP の選択になる。
CP（整合性 + 分断耐性）

分断時は応答を止めてでも整合性を守る。

例：HBase, Zookeeper, MongoDB（設定次第）

AP（可用性 + 分断耐性）

分断時も応答し続けるが、古いデータを返す可能性がある（結果整合性）。

例：Cassandra, DynamoDB, CouchDB

CA（整合性 + 可用性）

単一ノードなら両立できるが、分散環境では現実的に選べない。

例：単一ノードの RDBMS

まとめ

SQL vs NoSQL

整合性・スキーマの安定性 → SQL。
スケール・柔軟性・大規模アクセス → NoSQL。
用途に応じて使い分ける（ポリグロット）。

インデックス

読み取りを速くする目次。
書き込みコストとのトレードオフを意識し、
EXPLAIN で必要なものだけ張る。

レプリケーション

同じデータを複製して読み取り分散・高可用性を実現。
書き込みスケールには効かない。

シャーディング

データを分割して書き込みスケールを実現。
複雑さが増すため、本当に必要になってから導入する。
